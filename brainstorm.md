# rust-agent 核心理念与设计蓝图

> 本文档记录 rust-agent 的立足之本：权重驱动的优先级执行 + 主动苏格拉底式高密度提问思考。
> 状态：讨论中，尚未进入编码阶段。
> 可执行规格见：`workflow-spec.md`

---

## 一、两大核心理念

### 1. 权重系统（源码级，不可跳过）

**核心原则**：权重必须写进源码，通过 Rust 代码硬编码规则驱动 AI 分配权重，而不是通过提示词。提示词会偷懒，源码不会。

**设计要点**：
- 每个任务动态分配权重，由 AI 根据源码规则自动计算
- 权重决定开发方向和先后顺序
- 权重高的任务优先执行
- 权重评估器是 Rust 代码，不是提示词模板

**为什么不用提示词**：
- 提示词容易被 LLM "忽略"或"简化"
- 源码级规则是强制执行的，AI 必须遵守
- 可以精确控制权重计算的每一步

### 2. 高密度提问式思考（主动模式）

**核心原则**：AI 必须主动提问、主动思考，而不是被动等待用户引导。用户在编程时不会想那么多，所以 AI 要自己做好思考工作。

**设计要点**：
- 主动模式：用户提出需求后，AI 自动进入思考流程
- 问题由 AI 根据上下文动态生成，不是固定模板
- 递进式思考：问题 A → 问题 B → 问题 C → ...，不断深入
- 在思考过程中不断给出答案，形成闭环

**与现有苏格拉底引擎的区别**：
- 现有：被动模式，用户手动调用 `socratic_analyze` 工具
- 目标：主动模式，系统自动触发，用户只需提出需求

## 二、总体工作流

```
用户提出编程需求
        │
        ▼
┌─────────────────────────────────────┐
│ 阶段 1：高密度提问思考（Socratic）    │
│                                     │
│  遇到难题/复杂需求 → 自动多提问题    │
│  → AI 根据上下文动态生成问题         │
│  → 递进式深挖（问题链 A→B→C→...）   │
│  → 每个问题 AI 自己回答              │
│  → 想明白问题的本质                  │
└─────────────────────────────────────┘
        │
        ▼
┌─────────────────────────────────────┐
│ 阶段 2：列计划                        │
│                                     │
│  基于思考结果生成执行计划             │
│  → 拆解为具体步骤                    │
│  → 识别依赖关系                      │
└─────────────────────────────────────┘
        │
        ▼
┌─────────────────────────────────────┐
│ 阶段 3：源码级权重分配                │
│                                     │
│  每个步骤由 AI 按源码规则计算权重     │
│  → 风险因子：风险越高，权重越高      │
│  → 依赖因子：有未完成依赖，权重降低  │
│  → 复杂度因子：越复杂，权重越高      │
│  → 按权重排序执行队列                │
└─────────────────────────────────────┘
        │
        ▼
┌─────────────────────────────────────┐
│ 阶段 4：按权重执行                    │
│                                     │
│  取最高权重的步骤                    │
│  → 执行前再次苏格拉底思考（微观推理）│
│  → 执行 → 验证 → 更新权重            │
│  → 如果太复杂，递归拆分              │
│  → 循环直到全部完成                  │
└─────────────────────────────────────┘
        │
        ▼
┌─────────────────────────────────────┐
│ 阶段 5：汇总与保存                    │
│                                     │
│  生成执行报告                        │
│  → 保存学习成果到记忆系统            │
└─────────────────────────────────────┘
```

## 三、关键设计决策

### 3.1 权重粒度：每个任务动态分配

- **不是**按开发阶段（分析/设计/实现/测试）固定分配
- **不是**人为预设权重值
- **是**每个具体任务在执行前，由 AI 根据源码规则动态计算

**权重评估维度（源码硬编码）**：

| 维度 | 规则示例 | 权重影响 |
|------|---------|---------|
| 风险 | 涉及文件写入、bash 执行、网络请求 | 风险越高 → 权重越高（需更仔细处理） |
| 依赖 | 有未完成的先决步骤 | 有依赖 → 权重降低（先做依赖） |
| 复杂度 | 代码行数预估、文件数量、API 调用数 | 越复杂 → 权重越高 |
| 影响面 | 修改的模块数量、是否为公共接口 | 影响越大 → 权重越高 |
| 回滚成本 | 修改是否可逆、是否有备份机制 | 成本越高 → 权重越高 |

### 3.2 苏格拉底问题：AI 动态生成

- **不是**固定模板（如现有的 6 种问题类型）
- **不是**用户手动输入问题
- **是**AI 根据当前上下文（任务描述、代码状态、历史记录）动态生成针对性问题

**问题生成策略**：
1. 分析任务描述，提取关键词和潜在难点
2. 识别代码库中与任务相关的文件/模块
3. 根据已有知识（记忆系统）推断可能的陷阱
4. 生成递进式问题链，每个问题基于前一个答案

**示例**：
```
用户：帮我给这个项目加上用户认证

AI 自动思考：
  Q1: 这个项目的架构是什么？是前后端分离还是单体应用？
  A1: 分析项目结构 → 是前后端分离的 React + Rust 后端

  Q2: 后端使用的 HTTP 框架是什么？是否有现成的 auth 中间件？
  A2: 检查 Cargo.toml → 使用 axum，有 tower-sessions

  Q3: 用户数据存储用什么？需要新增数据表吗？
  A3: 检查现有 DB  schema → 使用 PostgreSQL，需要新增 users 表

  Q4: 密码存储用什么哈希算法？是否需要支持 OAuth？
  A4: ...

→ 生成计划 → 分配权重 → 执行
```

### 3.3 递归拆分：LLM 自动判断

- **不是**硬编码的复杂度阈值（如 "超过 100 行就拆分"）
- **是**LLM 根据任务语义判断是否太复杂/太难

**递归触发条件（LLM 判断）**：
- 任务涉及多个不相关的子领域
- 任务需要修改大量文件
- 任务有高风险操作且缺乏明确方案
- 任务的依赖关系过于复杂

**递归流程**：
```
执行任务 X
  → LLM 判断：X 太复杂
  → 回到阶段 1：对 X 进行苏格拉底提问
  → 生成子计划 X1, X2, X3
  → 为 X1/X2/X3 分配权重
  → 按权重执行 X1/X2/X3
  → 完成后回到 X 的执行
```

## 六、待讨论问题（已讨论并确认）

### 1. 权重规则的维护方式

**结论：源码硬编码 + 环境变量微调，不接受配置文件。**

权重规则必须编译进二进制，确保 AI 强制执行不会"偷懒"。但给用户一个逃生舱——通过环境变量覆盖部分权重系数。

```rust
// weights.rs 中硬编码
const RISK_MULTIPLIER: f64 = 1.5;

// 允许环境变量覆盖
let risk_mul = env::var("PRIORITY_AGENT_WEIGHT_RISK_MUL")
    .ok()
    .and_then(|s| s.parse().ok())
    .unwrap_or(RISK_MULTIPLIER);
```

**原则**：规则逻辑在源码里不可变，系数可微调。

---

### 2. 苏格拉底 vs 直接模式切换

**结论：设置"快速通道"，由 LLM 做轻量级语义判断，不是硬编码命令名匹配。**

在 `ConversationLoop` 闸门处，LLM 先做一个低成本判断（1-2 轮）：

| 类型 | 示例 | 处理方式 |
|------|------|---------|
| 简单查询 | `/help`、`/clear`、`git status` | 直接走现有对话模式 |
| 复杂任务 | "重构这个模块"、"加 auth 系统" | 进入 WorkflowEngine |

**判断标准（语义级）**：是否涉及多文件修改？是否有架构决策？是否有风险操作？

---

### 3. 自问自答的 token 成本

**结论：三层控制 + 优雅降级。**

| 层级 | 控制方式 | 默认值 |
|------|---------|--------|
| 宏观 | 每轮对话最多触发 1 次苏格拉底流程 | 硬编码 |
| 中观 | 单次思考最多 N 轮 Q&A | `PRIORITY_AGENT_SOCRATIC_MAX_ROUNDS=5` |
| 微观 | 单个问题答案最多 M tokens | `PRIORITY_AGENT_SOCRATIC_ANSWER_BUDGET=500` |

超过 budget 时：将未回答完的问题合并进执行计划，交给执行阶段"边做边想"，不硬撑。

---

### 4. 递归深度限制

**结论：默认 3 层，可配置，底层强制平铺。**

```
L0: 用户需求 → 苏格拉底思考 → 计划
L1: 计划步骤太复杂 → 子计划 + 子思考
L2: 子步骤还复杂 → 孙子计划
L3: 到达底层，不再递归，直接平铺为原子操作
```

环境变量：`PRIORITY_AGENT_WORKFLOW_MAX_DEPTH=3`

到达 L3 时若 LLM 仍判断"太复杂"，WorkflowEngine 强制拆成原子操作，不无限递归。

---

### 5. 与现有 Plan Mode 的关系

**结论：WorkflowEngine 是 Plan Mode 的"主动增强版"，两者共存。**

| 特性 | Plan Mode (`/plan`) | WorkflowEngine |
|------|-------------------|----------------|
| 触发方式 | 用户手动调用 | AI 自动检测并触发 |
| 审批 | 每步人工审批 | 自动执行（可中断） |
| 权重排序 | 无 | 源码级权重计算 + 动态排序 |
| 苏格拉底 | 无 | 主动高密度提问 |
| 适用场景 | 重大架构变更 | 日常编程任务 |

两者底层复用同一个 `Plan` 数据结构，WorkflowEngine 多了"自动权重排序"和"自问自答"两个环节。
