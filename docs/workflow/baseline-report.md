## 1. 采样方法

### 1.1 样本来源

从项目真实 commit 历史（Phase 0–12，约 150+ 次 commit）中按分层抽样提取 20 个代表性任务：

| 复杂度 | 数量 | 选取标准 |
|--------|------|---------|
| 简单 | 5 | 单文件修改、无架构决策、低风险 |
| 中等 | 8 | 多文件协调、有依赖关系、局部设计决策 |
| 复杂 | 7 | 架构改动、跨模块、需递归拆分 |

### 1.2 标注维度

每个样本人工标注以下字段：

| 字段 | 说明 |
|------|------|
| `mainline_goal` | 该任务真正的主线目标（一句话） |
| `true_blocker` | 必须先完成才能推进的关键步骤 |
| `drift_risks` | 容易陷入的细枝末节/跑偏点 |
| `expected_steps` | 预期应包含的关键步骤清单 |
| `complexity` | 简单 / 中等 / 复杂 |
| `estimated_files` | 预计修改文件数 |

## 2. 四硬指标测量方法

### 2.1 主线命中率（Mainline Hit Rate）

```
mainline_hit_rate = 命中主线的任务数 / 总任务数

"命中主线"定义：
- AI 的首个执行步骤 == 人工标注的 true_blocker
- 或首个步骤直接服务于 mainline_goal（评审判定）
```

### 2.2 漂移打断率（Drift Interruption Rate）

```
drift_interruption_rate = 用户主动打断次数 / 总任务数

"主动打断"定义：
- 用户说"跑偏了"/"这不是重点"/"先做这个"
- 或用户手动跳过了 AI 建议的 2 个以上步骤
```

### 2.3 首轮计划覆盖率（First Plan Coverage）

```
first_plan_coverage = (首轮计划步骤 ∩ 人工 expected_steps) / |expected_steps|

"首轮计划"定义：
- WorkflowEngine 第一次进入 EXECUTING 前的计划快照
```

### 2.4 返工率（Rework Rate）

```
rework_rate = 返工次数 / 总任务数

"返工"定义：
- 同一步骤因失败/错误被重新执行
- 或执行后因方向错误被撤销重做
```

## 4. 当前基线（无 WorkflowEngine，即现有 Direct Mode）

基于对过去 20 个真实任务的复盘（人工评审 AI 当时的执行记录）：

| 指标 | 当前值 | 备注 |
|------|--------|------|
| 主线命中率 | ~35% | AI 经常从边缘步骤开始，而非 blocker |
| 漂移打断率 | ~35% | 约 1/3 任务需要用户纠偏 |
| 首轮计划覆盖率 | ~55% | 经常遗漏关键步骤，后续补做 |
| 返工率 | ~1.8 | 平均每任务返工 1.8 次 |

### 4.1 根因分析（人工判定）

1. **无主线意识**：AI 按工具调用便利性排序，而非按业务关键路径
2. **不主动提问**：用户不提就不问，导致假设错误后续返工
3. **计划不完整**：首轮计划平均遗漏 3-4 个关键步骤
4. **无权重排序**：同等对待所有步骤，风险高的没有优先处理

## 7. 复现方法

要复现本基线：

1. 读取 `samples.json` 中任意样本
2. 将 `task_description` 作为用户输入提交给 AI
3. 记录 AI 的实际执行步骤
4. 与 `expected_steps` / `true_blocker` / `mainline_goal` 对比
5. 按第 2 节公式计算 4 项指标
