# 面向中国市场的桌面编程 Agent 项目策划书

日期：2026-05-21

## 1. 项目摘要

本项目计划打造一款面向中国普通用户的桌面 AI 项目生产应用。用户不需要会编程，也不需要安装命令行工具、配置海外账号或理解代码仓库，只需要用中文描述目标，应用即可帮助用户创建、修改、运行和验证一个真实项目。

产品形态不是传统 IDE，也不是面向程序员的 CLI，而是一个可安装的 Windows 和 macOS 桌面应用。用户看到的是类似 Codex / Claude Code 对话体验的聊天界面、任务进度、运行预览和结果交付，不默认展示项目代码。代码、依赖安装、终端命令、文件结构、验证和修复过程由本地 Agent runtime 在后台完成。

一句话定位：

> 中国普通用户可用的 AI 项目制作桌面应用，让不会编程的人也能用中文做出可运行的小软件、网站、自动化工具和业务原型。

## 2. 为什么现在值得做

中国市场的机会不在于再做一个通用聊天机器人，而在于把 AI 编程能力从“开发者工具”变成“普通人可使用的项目生产工具”。

现有海外 AI 编程工具能力很强，但对中国普通用户有明显门槛：账号、网络、支付、英文文档、命令行安装和 API 配置都不适合大众使用。国内大模型供给已经足够丰富，DeepSeek、Kimi、GLM、通义千问、MiniMax 等模型都提供了 API 或平台接入能力，但面向普通人的完整桌面项目生产体验仍然缺失。

更具体地说，现在中国普通用户接触 AI 编程能力主要卡在四个地方：

- 海外 Claude Code / Codex 能力强，但官方入口主要是终端、IDE 或 GitHub 工作流，中国用户还会遇到访问、账号、支付和 API 可用性门槛。
- GitHub 上的开源 AI 编程项目很多，但普通用户很难稳定访问和使用 GitHub，也不会 clone 仓库、装依赖、读英文 README 或处理终端报错。
- 国内大模型厂商大多提供的是 LLM、API、控制台或云平台，模型能力没有自动变成普通用户能直接完成项目的软件。
- 国内已有编程工具也多是 CLI、TUI 或 VS Code 插件，默认用户懂命令行、项目目录和 API Key，对非编程背景用户仍然门槛很高。

因此，本项目真正要做的不是再提供一个模型入口，而是提供一个开箱即用的 AI 编程软件：下载安装后，用户用中文聊天就能创建、运行、修改和交付项目。

公开资料也能看到一个现实分层：

- 阿里云百炼已经提供千问及第三方模型服务，并列出 DeepSeek、Kimi、GLM、MiniMax 等模型接入，说明国内模型供给侧正在成熟。
- Qwen Code、Crush 等国内工具仍主要是终端或开发者工具，安装和配置路径仍要求用户进入终端、配置 API Key 或 settings 文件。
- Kimi、GLM、Qwen-Coder 等模型都在强调 Agent、Coding、工具调用和长上下文能力，但模型能力本身还没有自动转化成普通用户能直接完成项目的产品。

因此本项目的核心机会是：用桌面应用把模型能力、项目环境、文件系统、验证命令和交付流程封装起来，变成普通用户能理解和信任的“项目助理”。

## 3. 目标用户

第一阶段不面向专业程序员抢 IDE 市场，而面向“有想法但不会写代码”的用户：

- 个体创业者：想快速做落地页、小工具、订单管理、数据看板、客户表单。
- 中小商家和运营人员：想做活动页面、商品展示页、内部自动化脚本、表格处理工具。
- 学生和研究人员：想做课程项目、数据可视化、小型实验工具。
- 自媒体和内容团队：想做互动页面、资料整理工具、内容生成工作流。
- 产品经理和设计师：想把原型从“图”推进到“能跑的 demo”。

对这类用户来说，核心诉求不是“给我代码”，而是：

- 我说清楚想法，你帮我做出来。
- 做出来之后我能看到、能试用、能修改。
- 出错时你能自己修，而不是让我看一堆报错。
- 我不需要知道 npm、cargo、pip、git、终端是什么。

## 4. 产品定位

建议对外定位为“AI 项目工作台”，而不是“AI 编程助手”。

原因是“编程助手”默认目标用户是程序员，而我们真正要打开的是非程序员市场。产品可以拥有强编程能力，但前台叙事应该是“做项目”和“交付结果”。

推荐产品名称占位：

- 项目代号：Priority Studio
- 中文描述：AI 项目工作台
- 英文描述：Desktop AI Project Studio for China

核心差异：

| 维度 | 传统 AI 编程工具 | 本项目 |
| --- | --- | --- |
| 入口 | CLI / IDE / 插件 | 可安装桌面应用 |
| 用户 | 程序员 | 普通用户、轻技术用户、创业者 |
| 展示 | 代码、diff、终端日志 | 聊天、预览、进度、结果 |
| 模型 | 海外模型为主 | 国内模型优先，多模型路由 |
| 环境 | 用户自己配置 | 应用托管项目环境和验证流程 |
| 成功标准 | 代码生成质量 | 用户能否得到可用项目 |

## 5. 核心产品体验

### 5.1 第一次打开

用户下载并安装应用后，只需要完成三件事：

1. 选择使用方式：内置套餐、绑定自己的国内模型账号、企业团队账号。
2. 选择项目类型：网页、小程序原型、自动化工具、数据看板、文档/表格工具、其他。
3. 直接输入需求：例如“帮我做一个给咖啡店用的会员积分小程序原型”。

不要要求用户先理解代码仓库、终端、Node.js、Python、Git 或 API Key。

### 5.2 主界面

主界面应该是聊天驱动，但不是单纯聊天：

- 左侧：项目列表和历史会话。
- 中间：对话、任务进度、关键选择。
- 右侧或弹层：运行预览、截图、文件导出、发布入口。
- 底部：输入框，支持文字、图片、文件、表格。

默认不展示代码。需要时提供“高级模式”，允许懂技术的用户查看文件、日志、命令和 diff。

### 5.3 项目生成流程

用户输入需求后，Agent 不应该直接输出代码，而应该完成一条闭环：

1. 理解需求，必要时问少量关键问题。
2. 创建本地项目目录。
3. 选择技术栈和模板。
4. 写入文件。
5. 安装依赖。
6. 启动本地预览。
7. 自动检查错误。
8. 自动修复失败。
9. 向用户展示可用结果。
10. 允许用户继续用中文修改。

对普通用户来说，每一步只需要看到“正在搭建项目”“正在安装依赖”“正在修复启动错误”“预览已就绪”这种状态，不需要看到底层命令。

### 5.4 修改和迭代

用户不应该被迫进入代码编辑器。典型修改方式：

- “把主色调改成绿色。”
- “加一个会员等级。”
- “这个按钮太小了。”
- “把表格导出成 Excel。”
- “帮我做成可以发给客户看的版本。”

Agent 后台执行代码变更和验证，前台只展示变化说明、预览和可确认的结果。

### 5.5 交付方式

第一阶段可以支持四种交付：

- 本地运行：用户在应用里打开项目。
- 导出压缩包：交给开发者或客户。
- 静态网站发布：一键发布到托管服务。
- 项目快照：保存当前版本，可回退。

后续再扩展到小程序、服务器部署、企业内网部署和团队协作。

## 6. 当前项目的技术资产

现有 `priority-agent` 项目已经具备一部分很有价值的底层能力，不应该推倒重写。

当前可复用资产：

- Rust runtime：适合作为桌面应用的本地后台核心。
- 多 Provider 接入：已有 Kimi / MiniMax / OpenAI / OpenAI-compatible / Custom 等 provider 类型，可扩展到 DeepSeek、GLM、Qwen 等国内模型。
- 工具执行系统：具备文件读写、搜索、命令执行、权限、验证和恢复相关能力。
- 运行追踪：已有 turn trace、tool record、validation closeout 等证据体系。
- 记忆和检索：适合沉淀用户偏好、项目历史、常用技术栈和修复经验。
- 权限与安全：已有工具权限、风险信号和审批逻辑，可迁移成桌面应用里的“允许这次操作”“信任此项目”等交互。
- 验证闭环：已有真实项目 gauntlet 和 required validation 思路，适合把“能运行”作为产品硬指标。

这意味着桌面版的正确做法不是先写一个漂亮壳，而是把现有 runtime 变成后台项目执行引擎，再做一个非程序员可理解的前台。

## 6.1 当前开发进展

截至 2026-05-21 的 `docs/PROJECT_STATUS.md` 记录，`priority-agent` 已经不是停留在想法阶段，而是一个可工作的 Rust 编程 Agent runtime。它当前的主要进展包括：

- 交互式 coding CLI：已经具备类 Claude/Codex 的本地对话式编码入口。
- 状态化 runtime：包含 intent routing、turn trace、session goal、recovery plan、required validation closeout。
- 工具执行系统：支持文件读写、搜索、bash/terminal、验证命令、工具结果记录和失败恢复元数据。
- 权限和风险控制：已有权限规则、风险信号、approval response、permission denial recovery 和 trace 证据。
- 记忆和上下文：已有 memory namespace search、conflict hints、retrieval context、记忆行为断言和产品成熟度测试。
- 多模型 provider：已有 Kimi、MiniMax、OpenAI-compatible、Custom 等 provider 架构，可扩展到 DeepSeek、GLM、Qwen、百炼等国内模型。
- 评测和验证：项目状态文档记录最新 deterministic local tests 为 `1468 passed; 0 failed`，real-project-coding gauntlet 为 `15/15 passed`。

对合作方来说，这说明本项目不是从零开始做一个桌面壳，而是已有后台 Agent 执行底座。下一步的关键是产品化：把 runtime 迁移到可安装桌面应用里，让普通用户无需理解 CLI、代码和终端。

## 6.2 尚未完成的产品化工作

当前项目的短板也需要诚实说明：现有能力主要在后台 runtime 和 CLI，面向普通用户的桌面产品还没有完成。后续应优先补齐：

- Tauri 桌面壳：Windows/macOS 可安装应用。
- 聊天 + 预览界面：隐藏代码和命令，只展示项目进度、运行预览和交付结果。
- 项目工作区和沙箱：每个项目独立运行，支持快照、回退和权限边界。
- 国内模型配置/套餐：用户不需要自己处理复杂 API Key 和 provider 配置。
- 非程序员 onboarding：示例项目、模板、少量关键问题引导用户完成第一个项目。
- 内测数据闭环：记录任务完成率、失败原因、平均成本、用户修改轮次和付费意愿。

## 7. 建议技术架构

推荐采用“桌面壳 + 本地 Rust Agent 引擎 + 多模型网关”的架构。

### 7.1 桌面应用层

建议优先考虑 Tauri：

- 当前核心是 Rust，Tauri 能直接复用 Rust 能力。
- Windows 和 macOS 可统一打包。
- 包体相对 Electron 更轻。
- 更适合本地文件、命令、权限和安全边界管理。

前端可使用 React / Vue / Svelte 中任一种，重点不是框架，而是把复杂工程过程隐藏成清晰的聊天、进度和预览体验。

### 7.2 Agent Runtime 层

Rust runtime 继续负责：

- 会话管理。
- 项目目录和文件操作。
- 命令执行和进程管理。
- 权限审批。
- 工具调用。
- 模型请求。
- 失败诊断和自动修复。
- 验证命令和运行预览。
- 项目快照和回滚。

### 7.3 模型层

模型策略必须 provider-neutral。不要把产品绑定到任何单一模型。

第一阶段建议接入：

- DeepSeek：作为通用和性价比模型。
- Kimi：长上下文、Agent、代码和多模态方向。
- GLM：Agentic Coding、多模态 Coding、国产生态合作方向。
- Qwen / Qwen-Coder：代码模型、OpenAI 兼容生态、阿里云百炼渠道。
- MiniMax：作为备选高性价比和多模态供应。

模型路由策略：

- 简单聊天和需求澄清：低成本模型。
- 代码生成和大范围修改：代码能力强的模型。
- 长项目上下文：长上下文模型。
- 前端视觉复刻：多模态 Coding 模型。
- 验证失败修复：更强推理模型。
- 预算敏感用户：优先低成本和缓存策略。

### 7.4 安全与隔离

普通用户桌面产品必须把安全放在前面：

- 每个项目在独立工作区运行。
- 默认禁止修改项目目录外文件。
- 安装依赖、启动服务、删除文件、联网发布都需要清晰权限边界。
- 后台命令默认不可见，但必须可审计。
- 对外发布前需要做敏感信息扫描。
- 所有关键操作保留可回退快照。

## 8. MVP 范围

MVP 不要试图覆盖所有软件类型。第一版应该聚焦“能稳定做出小型 Web 项目和自动化工具”。

### 8.1 MVP 必须有

- Windows / macOS 可安装包。
- 中文聊天式项目创建。
- 隐藏代码的项目工作区。
- 本地项目生成、修改、运行、预览。
- 国内模型接入：至少 2 到 3 个 provider。
- 自动安装依赖和启动开发服务器。
- 自动验证和失败修复。
- 项目历史、快照和回退。
- 导出项目压缩包。
- 基础计费或 API Key 绑定方案。

### 8.2 MVP 不要做

- 不要一开始做完整 IDE。
- 不要一开始做团队协作。
- 不要一开始做手机端。
- 不要一开始做所有语言和所有框架。
- 不要把代码编辑器作为主体验。
- 不要承诺替代专业程序员。

## 9. 6 个月路线图

### 第 1 阶段：桌面壳和 Runtime 打通，1 到 2 个月

目标：把现有 CLI runtime 接进桌面应用。

交付：

- Tauri 桌面应用骨架。
- 聊天界面。
- 本地 project workspace。
- Rust runtime 后台调用。
- 一个 Web 项目模板闭环。
- provider 配置界面。
- 基础日志和错误展示。

验收标准：

- 用户输入“做一个个人作品集网站”，应用能创建项目、运行预览、修改一次样式并成功展示。

### 第 2 阶段：非程序员 MVP，2 到 3 个月

目标：让普通用户能完成真实小项目。

交付：

- 多项目管理。
- 运行预览面板。
- 自动依赖安装。
- 自动验证和修复。
- 快照和回退。
- 导出 zip。
- 2 到 3 个国内模型 provider。
- 20 个真实用户任务测试集。

验收标准：

- 20 个典型小项目任务中，至少 14 个能在不展示代码的情况下完成首版可运行结果。

### 第 3 阶段：封闭测试和商业化验证，1 到 2 个月

目标：验证用户愿不愿意持续使用和付费。

交付：

- 内测版本。
- 用户引导和示例项目。
- 模型成本监控。
- 简单套餐或额度系统。
- 崩溃和失败反馈上报。
- 项目发布或托管实验。

验收标准：

- 50 到 100 名真实内测用户。
- 留存、任务完成率、平均模型成本、失败原因有可观测数据。
- 找到 2 到 3 个高频付费场景。

### 后续开发计划拆解

建议按 0-180 天推进：

| 时间窗口 | 开发重点 | 可交付结果 |
| --- | --- | --- |
| 0-30 天 | Tauri 项目骨架、聊天前台、Rust runtime 后台调用、项目工作区 | 用户输入一句中文需求后，能创建并打开一个本地 Web 项目 |
| 31-60 天 | 运行预览、依赖安装、失败诊断、自动修复、快照回退、导出 zip | 完成个人作品集、活动页、本地工具三类模板闭环 |
| 61-120 天 | 接入 2-3 个国内 provider，模型路由、成本记录、非程序员 onboarding | 20 个典型任务中至少 14 个完成首版可运行结果 |
| 121-180 天 | 50-100 名内测用户、套餐/额度、失败上报、用户访谈和场景复盘 | 找到 2-3 个高频付费场景，形成下一轮融资/合作证据包 |

## 10. 商业模式

建议用混合模式：

- 免费试用：提供少量额度和示例项目。
- 个人订阅：按月付费，适合个人项目和学习。
- 项目包：按项目或按生成次数购买，适合低频用户。
- 团队版：多人项目、共享额度、企业模型账号。
- 企业私有化：对数据安全敏感客户提供本地或内网部署。

关键不是一开始最大化收入，而是尽早验证：

- 用户是否愿意用它完成真实项目。
- 用户在哪些场景反复使用。
- 单任务模型成本是否可控。
- 完成一个可用项目的平均成本和时间是多少。

## 11. 合作方需要看到的亮点

对外讲述时，重点不是“我们也能调用大模型”，而是这几个亮点：

1. 明确市场切口：中国普通用户还没有真正低门槛的 AI 项目生产桌面应用。
2. 差异化定位：不是开发者 CLI，不是 IDE 插件，而是隐藏代码的项目工作台。
3. 技术基础已有：现有 Rust Agent runtime 已有工具、权限、记忆、验证、追踪和多模型接入基础。
4. 国内模型优先：规避海外工具使用门槛，适配 DeepSeek、Kimi、GLM、Qwen、MiniMax 等国内供应。
5. 可验证产品观：成功标准不是“回答看起来不错”，而是项目是否能运行、能预览、能修改、能交付。
6. 可扩展商业路径：个人、团队、企业私有化、模型渠道合作都可演进。

## 12. 需要寻找的合作角色

优先找这几类人或资源：

- 桌面应用工程师：Tauri、Windows、macOS、安装包、自动更新。
- 前端产品工程师：聊天界面、项目预览、交互体验。
- AI Agent 工程师：工具调用、模型路由、失败修复、上下文管理。
- 国内模型渠道合作：百炼、智谱、Moonshot、DeepSeek、MiniMax 等 API 或套餐合作。
- 设计 / 产品合伙人：把复杂工程过程做成普通用户能理解的体验。
- 早期用户渠道：创业社群、电商运营、学生、设计师、产品经理、中小商家。

## 13. 主要风险和应对

| 风险 | 表现 | 应对 |
| --- | --- | --- |
| 模型能力不稳定 | 项目生成失败、修复循环 | 多模型路由、验证闭环、真实任务测试集 |
| 成本不可控 | 一次任务消耗大量 token | 上下文缓存、工具数量控制、分层模型路由、预算提示 |
| 普通用户需求太发散 | 什么都想做，结果都做不好 | MVP 聚焦 Web 小项目和自动化工具 |
| 桌面环境复杂 | Windows/macOS 依赖安装失败 | 内置运行环境、模板化技术栈、清晰诊断 |
| 安全风险 | 命令误删文件、泄露密钥 | 项目沙箱、权限审批、快照回滚、敏感信息扫描 |
| 国内竞品跟进 | 大厂也做桌面入口 | 深耕项目完成率、用户记忆、本地工作流、垂直场景 |

## 14. 建议对外版本的一句话

> 我们正在做一款面向中国市场的桌面 AI 项目工作台。它不是给程序员用的命令行工具，而是让普通用户用中文聊天就能创建、运行、修改和交付小项目的应用。底层基于自研 Rust Agent runtime，支持国内多模型路由、自动验证、失败修复、项目快照和安全权限控制。我们的目标是把 AI 编程能力从“开发者工具”变成“普通人可用的项目生产力软件”。
