# LabRun Evidence Lineage And Concurrency Hardening Plan
Status: Implemented
Date: 2026-06-26

This plan follows the 2026-06-25 LabRun policy, audit, and supply-chain
hardening work. The next useful step is not another role or tool surface. It is
to make LabRun proof precise enough for long-running, parallel, multi-cycle work:

```text
cycle -> postdoc plan -> graduate task -> dispatch -> worktree
  -> validation command -> graduate result -> postdoc audit -> professor review
```

The review feedback is useful. Several items are already covered by the latest
policy hardening, but the core evidence-lineage concerns are real and should be
treated as the next P0 batch.

## Implementation Closeout - 2026-06-26

Implemented:

- Added `LabEvidenceProvenance` and persisted it on `GraduateResult`.
- Recorded source PostdocPlan IDs on generated GraduateTasks.
- Bound Lab validation events to run/cycle/plan/task/dispatch/agent task/result,
  command hash, verification root, worktree evidence, trust source, and
  `controlled_not_sandboxed` validation semantics.
- Stopped Postdoc audit from reusing LabRun-level recent validation events.
  Audit now filters validation proof by the current GraduateResult provenance
  and fails closed when validation belongs to another task or dispatch.
- Filtered Postdoc integration to the current cycle, latest accepted
  PostdocPlan, and GraduateTasks generated by that plan. Stale results are
  visible as excluded context but cannot enter current accepted output.
- Bound Graduate mutation permission to one explicit active GraduateTask and
  dispatch; multiple open tasks no longer merge scopes.
- Switched Postdoc code-aware audit to the Graduate verification root and added
  a missing-root fault test so the parent workspace is not silently substituted.
- Added explicit LabRun execution context, policy review state versions, and
  mutation-time state revalidation.
- Moved package-script workspace trust to a project-scoped trust store with an
  explicit environment override for tests/debugging.
- Split the mixed LabRun runtime helper file into:
  `artifact_factory.rs`, `graduate_verification.rs`, `postdoc_audit.rs`,
  `runtime_evidence.rs`, and a small `runtime.rs` transition helper.

## P0: Bind Validation Evidence To One Graduate Execution

### Target Invariant

A `PostdocIntegrationSummary` may only cite validation evidence that is
proven to belong to the exact `GraduateResult` currently under review.

### Data Model

Add a reusable provenance object, for example:

```rust
pub struct LabEvidenceProvenance {
    pub lab_run_id: String,
    pub cycle_id: Option<String>,
    pub source_postdoc_plan_artifact_id: Option<String>,
    pub graduate_task_id: Option<String>,
    pub dispatch_id: Option<String>,
    pub agent_task_id: Option<String>,
    pub graduate_result_artifact_id: Option<String>,
    pub verification_root: Option<String>,
    pub worktree_base_commit: Option<String>,
    pub worktree_head_commit: Option<String>,
    pub worktree_diff_hash: Option<String>,
    pub command_hash: Option<String>,
    pub verified_at: Option<DateTime<Utc>>,
}
```

This can start as a focused LabRun-internal model and be embedded in event
payloads before becoming a stable public artifact field.

### Required Changes

1. Extend validation event metadata.
   - Add optional `cycle_id`, `graduate_task_id`, `dispatch_id`,
     `agent_task_id`, `graduate_result_artifact_id`, `verification_root`,
     `worktree_head_commit`, and `command_hash`.
   - Keep backward compatibility for old validation events that lack these
     fields.

2. Add task-bound validation execution.
   - Introduce a function such as
     `run_lab_validation_commands_for_graduate_task(...)`.
   - Inputs should include `LabEvidenceProvenance`, not only `store` and
     `lab_run_id`.
   - Runtime verification should pass the active task and dispatch context into
     validation recording.

3. Stop using "latest validation events" as proof.
   - Replace `postdoc_validation_event_refs(store, lab_run_id)` with a query
     that filters by the current `graduate_result_artifact_id`, or by
     `graduate_task_id + dispatch_id + verification_root` before the result
     artifact exists.
   - If no bound validation event exists, the audit should mark the result
     `postdoc_audit_not_verified` or `postdoc_audit_needs_revision`.

4. Add negative tests.
   - Task A has passed validation.
   - Task B has no validation.
   - Postdoc audit for Task B must not reuse Task A's validation event.

## P0: Make GraduateResult Provenance Explicit

### Target Invariant

Every `GraduateResult` should be traceable to the plan, cycle, task, dispatch,
agent task, and worktree that produced it.

### Required Fields

Add explicit provenance to `GraduateResult` or the artifact envelope metadata:

```text
cycle_id
source_postdoc_plan_artifact_id
graduate_task_id
dispatch_id
agent_task_id
worktree_ref
verification_root
verified_at
validation_event_ids
```

`GraduateTask` already stores `cycle_id`; `GraduateDispatchRecord` already
stores `dispatch_id` and `task_id`. The missing part is carrying those IDs into
`GraduateResult`, validation events, and Postdoc audit queries.

### Required Changes

1. When creating a graduate task from a Postdoc plan, persist
   `source_postdoc_plan_artifact_id`.
2. When dispatching a graduate task, record `dispatch_id` and `agent_task_id`
   in the eventual `GraduateResult`.
3. When syncing durable graduate output, bind the generated `GraduateResult`
   to the latest matching dispatch and verification root.
4. If a result lacks provenance, treat it as legacy/unverified unless an
   explicit migration or compatibility path marks it safe.

## P0: Filter Postdoc Integration To The Current Cycle And Plan

### Target Invariant

At `postdoc_review`, integration must include only results from:

```text
current LabRun
current cycle
current accepted PostdocPlan
GraduateTasks generated by that plan
verified GraduateResults for those tasks and dispatches
```

### Required Changes

1. Find the current accepted `PostdocPlan` artifact for the current cycle.
2. Build the expected graduate task set from that plan's task provenance.
3. Load only `GraduateResult` artifacts whose provenance matches:
   - same `lab_run_id`;
   - same `cycle_id`;
   - same `source_postdoc_plan_artifact_id`;
   - one of the current plan's graduate tasks;
   - a verified dispatch/result binding.
4. Exclude superseded, previous-cycle, or unbound results from accepted output.
5. Include an explicit audit line for excluded stale results so debugging is
   visible but old results cannot pollute the current review.

### Tests

- Cycle 0 has two verified GraduateResults.
- Cycle 1 has one verified GraduateResult.
- Cycle 1 Postdoc integration must include only the Cycle 1 result.
- A superseded GraduateResult must remain visible in trace/proof, but not in
  `accepted_results`.

## P0: Bind Graduate Scope To One Active Task

### Target Invariant

Graduate write permission is granted to exactly one active task and dispatch,
not to the union of all open graduate tasks.

### Required Context

Action review or tool context should carry:

```text
active_lab_run_id
active_graduate_task_id
active_dispatch_id
lab_mode_enabled
lab_state_version
```

### Required Changes

1. Extend action review input with optional LabRun execution context.
2. Change `review_graduate_mutation()` to prefer the explicit active graduate
   task ID.
3. Remove the "all open tasks" fallback for mutation allowance.
4. If no active task is present, return `ask` or `deny` for mutation rather than
   merging scopes.
5. Keep read-only actions allowed.

### Tests

- Task A allows `src/api`.
- Task B allows `src/memory`.
- Active task is A.
- A mutation under `src/api` is allowed.
- A mutation under `src/memory` is denied even though Task B is open.

## P0: Postdoc Audit The Same Worktree That Was Verified

### Target Invariant

Graduate verification and Postdoc code-aware audit must inspect the same code.

### Preferred Flow

```text
Graduate verified in isolated worktree
  -> Postdoc audits exact verification_root/worktree
  -> Postdoc accepts or requests revision
  -> merge or report according to policy
```

### Required Changes

1. Store `verification_root`, base commit, head commit, and diff hash in the
   graduate result provenance.
2. Change Postdoc audit to read snippets and diffs from `verification_root`
   when provenance points to an isolated worktree.
3. Use parent workspace only when the provenance explicitly says the result was
   verified in the parent workspace or after a successful merge proof.
4. Record audit payload fields:
   - `audit_root`;
   - `verification_root`;
   - `worktree_base_commit`;
   - `worktree_head_commit`;
   - `worktree_diff_hash`;
   - `task_id`;
   - `dispatch_id`.
5. If `verification_root` no longer exists, mark the audit as not verified and
   do not substitute the parent workspace silently.

## P1: Remove `latest_run()` From Hard Policy Decisions

### Target Invariant

The policy decision should be based on the LabRun context that initiated the
tool action, not whichever run happens to be latest on disk at review time.

### Required Changes

1. Add an explicit `LabRunExecutionContext`:

```rust
pub struct LabRunExecutionContext {
    pub lab_mode_enabled: bool,
    pub lab_run_id: Option<String>,
    pub lab_stage: Option<String>,
    pub lab_role: Option<LabRole>,
    pub lab_status: Option<LabRunStatus>,
    pub lab_state_version: Option<String>,
    pub active_graduate_task_id: Option<String>,
    pub active_dispatch_id: Option<String>,
}
```

2. Pass this context into action review and policy overlay.
3. Keep `latest_run()` only as a compatibility fallback for interactive Lab
   commands that explicitly operate on the latest run.
4. Add `lab_run_id`, `state_version`, `reviewed_stage`, `reviewed_owner`,
   `reviewed_status`, and `reviewed_at` to `LabRunPolicyReview`.

## P1: Close The Policy TOCTOU Window

### Target Invariant

The LabRun state reviewed for a mutation should be the same state used when the
mutation executes or is recorded.

### Required Changes

1. Add a lightweight state version, initially derived from persisted
   `updated_at` plus the active stage/owner/status.
2. `LabRunPolicyReview` stores that version.
3. Before executing a mutation, re-read the run by explicit `lab_run_id`.
4. If the version changed, fail closed with a retryable policy reason such as:
   `labrun_policy_state_changed`.
5. Record policy events against the reviewed `lab_run_id`, not `latest_run()`.
6. Longer term: add a lease or compare-and-swap helper in `LabStore` for
   state-changing commands.

## P1: Make Workspace Trust Project-Scoped

### Target Invariant

Trust belongs to a specific repository identity, not to the current process.

### Required Changes

1. Add a trust store such as:

```text
~/.priority-agent/trusted-workspaces.json
```

2. Trust records should include:

```json
{
  "canonical_path": "...",
  "repo_identity": "...",
  "repo_fingerprint": "...",
  "trusted_at": "...",
  "trust_scope": "package_scripts",
  "source": "user_approval"
}
```

3. Resolve workspace trust by canonical project root and git remote/fingerprint.
4. Keep `PRIORITY_AGENT_LAB_WORKSPACE_TRUST` only as an explicit test override.
5. Show trust source in `/lab proof`, `/permissions`, and validation events.
6. Unknown or untrusted workspaces should block package-script validation unless
   a sandboxed backend is available.

## P1/P2: Clarify Controlled Validation Semantics

Controlled validation currently means:

- no provider-authored `sh -lc`;
- no shell metacharacter injection;
- no path traversal outside the verification root;
- only allowlisted validation command families.

It does not mean:

- untrusted code isolation;
- network isolation;
- protection from Rust `build.rs`, proc macros, tests, Python imports, npm
  package scripts, or repository shell scripts;
- protection from host credentials in the process environment.

For untrusted workspaces, the future backend should run in an actual sandbox or
container with:

- restricted filesystem mount;
- clean environment;
- network disabled by default;
- CPU, memory, process, and timeout limits;
- no host credentials;
- explicit artifact export.

Until then, docs and proof output should describe the current runner as
`controlled_not_sandboxed`.

## P1: Split LabRun Orchestrator Runtime By Responsibility

The current file is under the source-size guard, but it mixes concerns. Split by
cohesion after the P0 provenance model is stable:

```text
src/lab/orchestrator/graduate_verification.rs
src/lab/orchestrator/postdoc_audit.rs
src/lab/orchestrator/runtime_evidence.rs
src/lab/orchestrator/artifact_factory.rs
```

Suggested boundaries:

- `graduate_verification.rs`: verification root resolution, workspace snapshot,
  required validation execution, changed-file scope checks.
- `postdoc_audit.rs`: code-aware audit, redaction, large-file omission,
  validation-event binding checks.
- `runtime_evidence.rs`: proof/event formatting, worktree evidence, command and
  diff hashes.
- `artifact_factory.rs`: stage artifact construction helpers that are not
  specific to one role.

## P1/P2: Persistence Concurrency And Fault Injection

Add tests that simulate:

- two processes or schedulers trying to advance the same run;
- user command and daemon updating the same run;
- crash after artifact write but before gate write;
- crash after dispatch success but before result binding;
- event append succeeds but state save fails;
- stale lease retry collides with a new dispatch;
- Postdoc audit root disappears after validation.

The goal is not to make all recovery automatic immediately. The first goal is
to detect and label partial states honestly.

## Definition Of Done

- Postdoc audit cannot reuse validation from a different task or dispatch.
- Postdoc integration cannot accept stale results from previous cycles or plans.
- Graduate mutation permission is bound to one active task and dispatch.
- Postdoc audit reads the same worktree/root that Graduate verification used.
- Policy review records explicit run/state context and fails closed on state
  changes before mutation.
- Workspace trust is stored and displayed per project/repository identity.
- Controlled validation proof says `controlled_not_sandboxed` unless a real
  sandbox backend is in use.
- Runtime helper modules are split by responsibility after tests protect the
  behavior.
